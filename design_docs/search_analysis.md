A design review of ai-workflows: sound bones, several load-bearing gaps
Executive summary
Verdict: the architecture is feasible, largely well-reasoned, and converges with mainstream patterns in enough places to be defensible. But it has three load-bearing gaps that will hurt before Milestone 3 ships, plus a cluster of smaller issues worth addressing now because they get expensive later. Specifically: (1) the static Planner-generated DAG will not survive contact with real workflows that need runtime fan-out sizing — every serious framework has moved to dynamic dispatch (Send in LangGraph, MediumLangChain events in LlamaIndex, LlamaindexPyPI orchestrator-worker subagents in the Claude Agent SDK); Claude API Docs (2) the bespoke SQLite checkpoint/resume will be coarser and less correct than you think, because the hard failure modes are mid-activity crashes, cancellation under asyncio.TaskGroup, and code changes between runs — all problems Temporal solved ten years ago; (3) there is no evaluation layer at all, which means you will be unable to answer "did my prompt change regress quality?" after the second workflow, and retrofitting that costs more than putting it in at M1.
The Primitives-Components-Workflows three-layer split is validated by AutoGen v0.4's nearly identical Core/AgentChat/Extensions split Jahanzaib +2 — keep it. The typed Component taxonomy (Worker, Validator, Fanout, Planner, Orchestrator, AgentLoop, HumanGate) is a genuine differentiator versus Haystack's undifferentiated @component and LangGraph's untyped nodes; it enables governance properties no mainstream framework offers. The YAML+prompts+Pydantic workflow artifact format is right. Tier-based routing by Component type is novel and worth keeping prominent.
The biggest scope question is honest: a solo developer with one-and-a-half real workflows is over-engineering a DAG orchestrator. CrewAI, LangGraph, and LlamaIndex all shipped DAG/graph orchestration before they had production customers — and all of them regret specific choices they were forced to make early. You are inheriting that risk for a use case (JVM modernization + side projects) that would be adequately served by Pydantic AI + pydantic-graph + pydantic-evals with maybe 400 lines of custom glue. If the goal is "learn how to build this class of framework," the architecture is defensible. If the goal is "ship JVM modernization fast," buying is cheaper than building and the first half of Milestone 1 should probably be deleted.
Feasibility analysis
Primitives soundness
The Primitives layer is the strongest part of the design. Three LLM adapters behind a Protocol is the right call — structural subtyping avoids Abstract Base Class ceremony and lets you test with stubs trivially. SQLite+WAL is reasonable for single-process, single-writer workloads, which is what you have. Separating tiers.yaml from pricing.yaml is cleaner than LangChain's string-identifier-plus-env-var sprawl. The stdlib-tools-as-Primitives choice (fs, shell, http, git) correctly treats them as shared capabilities rather than workflow-specific.
Real problems in Primitives:
The Protocol-based LLMClient will leak provider-specific behavior upward — this is a design fact, not a failure. Anthropic's prompt caching uses cache_control blocks that have no analog in Ollama; Anthropic's tool use has tool_use / tool_result content blocks while OpenAI has tool_calls and a separate tool role; Gemini has yet a third shape. Ofox A flat Message(role, content: list[ContentBlock]) is the right normalization target — Anthropic-shaped — but you need an explicit capability descriptor on each client (supports_prompt_caching, supports_parallel_tool_calls, supports_structured_output, max_context, supports_thinking). Otherwise components will either dispatch on isinstance(client, AnthropicClient) (layering violation) or assume features silently. Pydantic AI does this via the Model abstraction; PydanticAI you should steal it.
The prompt caching implementation is naive. "Auto-mark the last system block as cacheable" is a legitimate pattern but fails the most common cases: if your last system block contains templated {{variable}} substitutions (timestamps, run_id, workflow-specific context), the hash differs every call — you pay a cache write every time and read zero. claudeClaude API Docs Anthropic's recommended 2026 pattern uses up to 4 breakpoints: AnthropicAnthropic one on the last tool definition (1h TTL), one on the last truly static system block Anthropic (1h TTL), and top-level automatic caching for the growing messages history Pydantic (5m TTL). Claude API DocsAIHubMix You should validate your implementation by asserting cache_read_input_tokens > 0 on turn 2+ of any multi-turn run; if it isn't, caching is broken. AIHubMix
The sanitizer is theater. A regex list against prompt-injection patterns does not defend against adversarial content in a JVM repo. Simon Willison's 2024-2025 writing is unambiguous: pattern-matching defenses fail to novel phrasing trivially. Simon WillisonSubstack The realistic defenses are capability-based — tool allowlists per workflow (you have this), cwd restriction (you have this), dry_run (you have this), and the human gate on destructive operations (you have this). Delete the sanitizer or rebrand it as "logging for post-hoc forensics, not a security control." Keeping it as "defense" is worse than not having it because it creates false confidence. The real threat model for a solo dev on their own repos is low; the design should say so explicitly instead of pretending otherwise.
Retry policy is underspecified. 429/529-only retry is correct for server-side-transient errors, but you will also hit: APIConnectionError (transient network), Anthropic overloaded_error (should retry), Anthropic invalid_request_error (must not retry), tool-result validation errors from the model (these are "ModelRetry" territory — Pydantic AI's pattern of feeding validation errors back to the LLM PydanticAI is superior to hard failure). The policy needs a small taxonomy, not a single flag. Also: every framework with a serious retry story disables the underlying SDK's client retries (max_retries=0) when wrapping it, to prevent double-counting and accidental exponential amplification. Temporal Temporal's AI cookbook is explicit about this; Temporal your code should be too.
Migrations-as-manual-SQL will cause one specific class of bug. There is no rollback path and no version table. Use yoyo-migrations or sqlite-utils — both are one-file additions and give you a _migrations tracking table for free. Blue Book Alembic is overkill for SQLite. GitHub The cost of adding this in M1 is ten minutes; the cost of adding it in M5 after you have 50 runs' worth of data is a migration script.
Component model soundness
The component set is mostly minimal-correct. Specific observations:
AgentLoop versus Worker is a real distinction, not a smell. Worker = bounded (soft cap 15), pre-committed to producing a specific output shape. AgentLoop = open-ended, terminates on end_turn or done tool or max_iterations. Anthropic's own 2024-2026 guidance distinguishes these as "workflow step" versus "autonomous agent loop." AIMultiple Keep them separate. But the docs admit AgentLoop "violates the pure in intent rule," which tells me the rule is wrong, not AgentLoop. A Component is a typed unit of work with defined pre/post-conditions and a resumable checkpoint boundary — it does not need to be "pure." Revise the contract.
The two-phase Planner (Qwen explores → Opus plans) is sound but couples Planner to a specific model topology. This is fine for your fixed infrastructure but means Planner is not a generic component — it's TwoPhaseLLMPlanner. Make the phase count and model assignment configurable (PlannerConfig(explorers=[Qwen], planners=[Opus])), and treat a one-phase Planner (Opus with tool access, no Qwen) as the same class. Otherwise every refinement of the planning strategy requires editing components/planner.py.
Validator's structural-vs-semantic split is good and should be louder. CrewAI's task guardrails are nearly identical and they chain, where each validator's output feeds the next. Borrow the chaining: validators: list[Validator] rather than validator: Validator | None. Also: semantic validation by Haiku is cheap and correct for 90% of cases, but add an explicit LLMJudge(rubric, model) layered on Pydantic Evals semantics — it gives you evals and runtime validation from one abstraction.
Fanout with hard cap 8 and "double-failure = hard stop" is probably too conservative. Real map-reduce workloads (summarize 50 files, rewrite 30 tests) need fan-out widths of 20-50 with partial failure tolerance. The "one mitigation attempt then hard stop" rule is Anthropic-style "fail loud" — good for production, bad for batch refactors where you want a best-effort pass plus a retry list. Add a failure_policy: 'hard_stop' | 'best_effort' | 'n_of_m' on Fanout. LangGraph's Send + reducer pattern handles this cleanly; LangChain look at their defer=True node flag for "wait until all siblings finish, even failed ones." Medium
Orchestrator's one-mitigation-then-hard-stop policy is realistic for the JVM modernization use case and not realistic for most other workflows. Modernization has strong ordering and compile-gate invariants, so hard stops are the right default. Documentation generation, code review, migration audits are all best-effort by nature. The policy should be per-workflow, not global.
The critical Orchestrator gap is data flow between tasks (your open question C-14). This is not optional. Define it now. There are three reasonable patterns: (a) shared mutable state object with Pydantic schema and typed reducers (LangGraph model), (b) typed events passed between steps (LlamaIndex model), (c) explicit outputs: dict[task_id, artifact_id] with Pydantic schemas on task definitions (Haystack model with typed sockets). Recommendation: pattern (c). You already have Pydantic schemas per workflow; extend them to declare task inputs/outputs, let the Orchestrator type-check the DAG at plan time (reject plans where task B expects input task A doesn't produce), and store task outputs as artifacts in SQLite. Haystack's pipe.connect("a.out", "b.in") Haystack with compile-time validation Haystack is the model. Restack Skipping this will bite you in the first cross-task workflow.
Workflow layer soundness
Workflow-as-directory-with-workflow.yaml-plus-prompts/-plus-schemas/-plus-optional-custom_tools.py-plus-thin-run.py is the right shape and closely matches CrewAI's @CrewBase + config/agents.yaml + config/tasks.yaml pattern DeepWiki +2 that authors widely praise for readability.
The hard limit of YAML-as-workflow-definition is conditional logic and loops, which CrewAI tried to put in YAML and gave up, pointing users to its imperative Flows API instead. Your design is already correct here because orchestration lives in typed Components (Planner, Orchestrator) rather than YAML — keep it that way. Never add if / else / for to workflow.yaml. The moment you feel tempted, either add a new Component type or tell the user to write a second workflow.
Prompt templates with {{variable}} only — good for now, but watch for the moment you need partials or loops. CrewAI's {variable} interpolation CodeSignal suffices for 95% of workflows; CrewAI the remaining 5% invariably reach for Jinja2. When that happens, don't reinvent — just add Jinja2 with autoescape off, it's a 5-line change. Defer until demanded.
schemas/*.py as Pydantic is right. Make them required, not optional. Every workflow should declare InputSchema, OutputSchema, and per-task schemas. Without this, the Orchestrator can't type-check task wiring, Validators have nothing to validate against, and evals have no ground truth to compare to. This is the one place to be opinionated.
Import-linter for layer enforcement is the right tool. Add a CI check that imports from layer_3 in layer_2 fail, plus layer_2 in layer_1 fails. One .importlinter config file, one GitHub Action. This is table stakes for any framework claiming layered architecture.
Layering enforceability
The three-layer boundary is enforceable at the module level (import-linter will catch leaks) but not at the semantic level. Specifically:
Planner reaching into raw repo files via tools does not violate intent. Components using Primitives (tools) is exactly what the layering prescribes. What would violate intent is Planner reaching into another Component's private state or the Orchestrator reaching into a Worker's tool registry. Add _private prefixes and an import-linter rule to forbid cross-component peeks.
The real leak is that AgentLoop, Orchestrator, and Planner all need access to multiple Primitives at once (LLM client, tool registry, storage, cost tracker). This is fine, but it means these three components are structurally "thick" and the others ("Worker", "Validator", "Fanout", "HumanGate") are thin. Don't pretend they have equal weight. Consider a BaseOrchestrationComponent above BaseComponent for the three thick ones, so injected dependencies are explicit.
Technical / architectural gaps
Critical (fix before Milestone 1 ships):

Task-to-task data flow (C-14). Define it before writing the Orchestrator. Recommendation above: typed Pydantic schemas on each task declaring input and output, artifacts stored in SQLite, compile-time validation of the DAG against task schemas. Without this, the Orchestrator is incomplete.
Workflow versioning and reproducibility. If workflow.yaml changes between a run and a resume, the behavior is undefined today. Minimum viable discipline: store the full workflow directory's git SHA (or a content hash of all files in workflows/<name>/) in the runs table. On resume, refuse to continue if the hash differs — require explicit --force-workflow-version-mismatch. This is a 20-line feature. Without it, "resume" means "hope nothing changed."
Checkpoint granularity and cancellation semantics. Document what happens when the user hits Ctrl-C mid-asyncio.TaskGroup during a Fanout. Python 3.12's TaskGroup propagates CancelledError to all children, Python which is what you want, but you must write the checkpoint before launching children and mark success after all children either complete or fail atomically. Borrow LangGraph's superstep + pending-writes model: checkpoint at superstep boundaries, preserve the output of successful siblings even when one sibling fails, so resume doesn't re-run them. Substack +2 Without this you will re-do expensive LLM calls on restart.
Budget caps are not "out of scope" — they are a critical safety control. The user is paying out of pocket for Claude Max. A single runaway AgentLoop at Opus rates can burn $50 in a night. Add a max_run_cost_usd config that the cost tracker checks after every LLM call and raises BudgetExceeded at. This is a 30-line feature. If not in M1, it must be in M2 before AgentLoop ships.
Typed LLMClient capability descriptor. As above, add client.capabilities: ClientCapabilities with booleans for caching, parallel tool calls, structured output, thinking mode, max context. Components that need features check the capability, not the class.

Important (fix before Milestone 3 ships):

Observability beyond structlog. Emit OpenTelemetry spans with the experimental GenAI semantic conventions (gen_ai.operation.name, gen_ai.request.model, gen_ai.usage.*, gen_ai.response.model). OpenTelemetry Two lines via logfire.configure(); logfire.instrument_anthropic() or roll your own with opentelemetry-sdk. OTel GenAI conventions are experimental as of 2026 OpenTelemetry but broadly adopted (Langfuse, Phoenix, Braintrust, Logfire); DEV Community emitting them now costs nothing and buys trivial integration with every eval/obs tool.
Evaluation layer. Add pydantic-evals in Milestone 3 alongside the first workflow. Case + Dataset + LLMJudge is the minimum; PydanticAI ten cases per workflow catches the 80% of regressions. Without this, prompt iteration is blind and model swaps are terrifying.
Send-equivalent for runtime fan-out. Today the Planner generates the DAG up front, which cannot handle "search returned K docs, spawn K summarizers" unless K is known at plan time. Either (a) allow Workers to return a DAG fragment that the Orchestrator splices in, or (b) adopt LangGraph's Send pattern where a conditional edge returns a runtime-sized list of worker payloads. MediumMedium Static DAGs are strictly less expressive than both LangGraph and LlamaIndex today; the competition is a meaningful bar.
Pydantic v2 discriminated union on ContentBlock. The flat Message(role, content: list[ContentBlock]) schema needs ContentBlock = TextBlock | ToolUseBlock | ToolResultBlock with Field(discriminator='type') and a literal type field on each variant. Without the discriminator, Pydantic v2 will attempt each variant in order, produce confusing errors, and tank performance on long messages. Test it with a Message containing 50 tool-use blocks.
Schema migration framework. Swap manual SQL for yoyo-migrations (one file) GitHub or sqlite-utils.Database.migrate(). Adds a _migrations table and rollback paths. GitHub Ten-minute change in M1; a day in M5.

Nice-to-have:

Encrypted checkpoint serde. If you ever ship this or run it on a shared machine, plaintext SQLite containing prompts and LLM outputs is a leakage risk. LangGraph's EncryptedSerializer.from_pycryptodome_aes is the reference. Langchain Not urgent for solo-use.
Cross-run memory (a Store equivalent). Checkpoints are per-run. User/project facts ("this repo uses Spring Boot 2.7, JDK 17, Gradle") belong in a separate KV that survives runs. LangGraph's BaseStore is the model. Langchain Not urgent until workflow #3.
MCP server (deferred, correctly). Keep it deferred. MCP read-only over SQLite is useful for Claude-Desktop inspection of runs, but it's polish, not foundation.

Operational gaps
Debugging a 12-task DAG that fails at task 7. Current design: the user gets a "structured failure report" artifact and... what? Resume is stubbed; re-running costs 7 tasks. You need two specific affordances: (a) aiw inspect <run_id> --task 7 that prints the full prompt, tool calls, LLM output, and validator result for that task from SQLite; (b) aiw resume <run_id> --from-task 7 --override-prompt prompts/new.txt that re-runs from task 7 with an edited prompt without invalidating tasks 1-6. Both are CLI features over existing SQLite state. Without them, debugging is "read the log file and guess." LangGraph's time-travel (get_state_history + fork) is the gold standard; Pydantic Evals' span-based evaluators + Logfire's trace viewer is the pragmatic alternative. GitHub
Prompt iteration loop. Today: user edits prompts/foo.txt, re-runs the whole workflow. There is no diff view, no "rerun just this task with the new prompt," no A/B. Add aiw rerun-task <run_id> <task_id> that replays a single task from checkpointed inputs with current prompts. Combined with pydantic-evals cases, this is the minimum viable prompt-iteration loop. Without it, prompt work is slow and expensive.
Evaluation harness. Entirely missing. Until the user has aiw eval <workflow> that runs Cases against a Dataset and reports pass/fail + regression-vs-last-run, there is no principled way to answer "is my new Qwen exploration prompt better?" This is the single biggest unforced error in the design. Minimum implementation: one evals/ directory per workflow, pydantic-evals cases, aiw eval command that runs them and writes results to SQLite. Add in Milestone 3.
Ollama flakiness. The design assumes OllamaClient is always reachable. In practice Ollama OOMs on a Qwen2.5-Coder:32B hot-swap, the home desktop sleeps, VPN drops mid-call. You need (a) a health check at run start that fails fast with an actionable error, (b) per-provider retry with longer backoff for local LAN endpoints than for cloud APIs, (c) explicit handling of ConnectionError as "pause and surface" rather than "retry forever." Consider a fallback tier: if Ollama is unreachable, escalate exploration to Haiku automatically and log the escalation cost. This is one Escalator component away (deferred to Milestone 7, which is fine if you add the circuit-breaker primitive earlier).
Network reliability and in-flight runs. When the VPN drops during a run that currently has a cloud LLM call in flight: the call times out, the retry fires, possibly succeeds on the second attempt, all good — if you disabled the SDK's internal retries. If you didn't, you get double-counting and potential double side effects. Confirm max_retries=0 on every SDK client you wrap. Temporal Test this explicitly.
HumanGate 30-minute timeout default. Wrong for overnight batches. The design has resume-via-SQLite which mitigates, but state pileup is real. Recommendation: strict_review gates default to timeout=None (wait forever, user resumes); non-strict gates default to 30min. Or make it explicit per workflow. Also: the HumanGate state surviving a process restart is non-trivial — LlamaIndex's HITL approach has known bugs around FastAPI+websocket context restore; LlamaIndex follow Haystack's snapshot pattern (serialize full Component state to SQLite on pause, rehydrate on resume) GitHub rather than rolling your own.
Log rotation and disk usage. ~/.ai-workflows/runs/<run_id>/ will accumulate. Add a aiw gc --older-than 30d --keep-artifacts command, and either rotate structlog output per-run or use a single JSONL file per run with size caps. MLflow's mlruns/ directory has the same problem and no good answer; don't let that happen here.
Multi-run observability. Zero story today. After 50 runs, the user cannot answer "has Opus's planning accuracy drifted in the last month?" or "which workflow has the highest cost per run?" The llm_calls table with OTel spans + a simple SQL dashboard (or Logfire/Phoenix) solves this. One aiw stats --last 30d CLI command that SELECTs from runs, llm_calls, tasks is a one-afternoon feature that will pay back hundreds of hours over a year.
Product and scope gaps
The three-layer architecture is arguably over-engineered for one user with one-and-a-half workflows. Be honest about this. AutoGen v0.4 validates the shape of the layering; atal upadhyay +2 that does not mean you need it on day one. A pragmatic reduction: collapse Primitives and Components into one aiworkflows/ module with clearly named submodules, keep the Workflow layer as-is, and split back into Primitives/Components the day you have a second workflow that proves the boundary. You lose import-linter enforcement; you gain 30% less code. Given the user's own admission that this is their first build of this kind, this is probably the correct tradeoff.
DAG-from-day-one is premature. Milestone 3 (test_coverage_gap_fill) is almost certainly a linear pipeline: explore → generate → validate. Milestone 5 (slice_refactor) might be a real DAG. Committing to networkx + topological sort + checkpoint/resume in M1 for M3 is three months of scaffolding for a feature M3 does not need. Recommendation: in M1-M3, implement a linear Pipeline instead of a DAG Orchestrator. Swap to DAG in M4 when you build Orchestration Components. The Pipeline-to-DAG upgrade is mechanical; the DAG-from-day-one path risks shipping a Planner before you have seen a real plan. Haystack shipped linear pipelines first and added cycles only in 2.7, years later Haystack +3 — this is battle-tested sequencing.
The two-machine Ollama setup is fragile for marginal cost savings. Qwen2.5-Coder:32B on a LAN desktop saves maybe $5-20/day versus Haiku for exploration reads. It costs reliability and mental overhead. Recommendation: start with all-cloud (Haiku for exploration) in M1-M3. Add Ollama in M4 when exploration volume is provably high enough to matter. This also eliminates a whole class of early operational gaps (Ollama OOM, VPN drop, health checks) that would distract from core framework work. The user's stated goal is "flexible framework," not "cheap framework" — keep the framework-building path clear.
Milestone 1's 16 tasks will feel like infinite plumbing. The user predicts this themselves. Three ways to mitigate: (a) cut milestone 1 in half by deferring sanitizer (theater), cost tracker (ship post-M2 when real cost data exists), and aiw inspect/resume stubs (ship with M2 Components that need them); (b) write the first workflow as a pure script in M1 week 2, using only the LLM adapter and no other framework — this proves the adapter works and gives a concrete artifact to refactor against; (c) vendor as much as you can — pydantic-ai for the Agent abstraction eliminates AgentLoop work entirely, pydantic-evals eliminates evaluation work, pydantic-graph eliminates Orchestrator work. If you adopt all three, Milestone 1 shrinks to "Anthropic+Ollama adapters, Pydantic AI agent wrapper, SQLite run log, CLI scaffolding" — maybe five days of work instead of six weeks.
YAML-vs-Python has a consensus answer. LangGraph said no to YAML entirely. LlamaIndex said no to YAML for workflow topology. CrewAI tried YAML and hit a cliff at conditional logic. DigitalOcean Haystack uses YAML for serialization but authoring is Python. Haystack +2 The consensus: YAML for static config (roles, prompts, tools-by-name, schemas-by-name, model assignments), Python for control flow. Your design already matches this. Don't backslide.

Diverges: AutoGen's blocking UserProxyAgent for HITL is explicitly a footgun; your durable HumanGate is better. AutoGen removed built-in sequential workflows ("users glue with Python") — that's the anti-pattern your typed Components prevent; keep typed Components.
Borrow: (1) the Core/AgentChat/Extensions split as validation of your layering; (2) composable termination conditions as first-class objects (cond_a & cond_b | cond_c); (3) dump_component()/load_component() serialization of Components for debugging/replay.
Reject: blocking HITL; "glue with Python" as a substitute for typed orchestration primitives; multi-agent conversation as a primary pattern (AutoGen has moved away from chat-loop abstractions and so should you).
Note: AutoGen is effectively in maintenance as of late 2025; Microsoft is steering users to Microsoft Agent Framework. Don't invest in AutoGen-specific compatibility.
LlamaIndex Workflows
Diverges: event-driven @step versus your DAG.
Borrow: (1) type-hint-driven topology validation — at workflow load time, verify every task's output schema has a consumer; (2) Context.to_dict() full-state serialization as the resume model (your SQLite checkpoint should serialize enough to equal this); (3) InputRequiredEvent/HumanResponseEvent pattern for HumanGate (any step can request input, not only pre-declared gates). Your planner-generated DAG cannot handle "Worker discovers mid-run it needs approval" — Worker can't dynamically add a HumanGate. If that's required, either allow mid-run HumanGate insertion or accept the limitation and document it.
Reject: implicit event routing (hard to reason about at scale); no built-in durable checkpoint (you already plan SQLite — good).
Haystack 2.x
Converges: typed Components with run() signatures, DAG pipelines, Agent loop with exit_conditions.
Borrow: (1) compile-time type-checking on pipe.connect("a.out", "b.in") — this is the single most important pattern to steal for your DAG wiring; it catches mis-wired workflows before any LLM call; (2) Breakpoints + PipelineSnapshot / AgentSnapshot as the uniform pause/debug/HITL primitive; (3) ConditionalRouter + BranchJoiner as named primitives for branching and join; (4) PipelineComponentsBlockedError when no component can run — clear deadlock signal; steal the error.
Reject: uniform untyped Components (your typed taxonomy is better); implicit cycle semantics with visit counters (make your loops explicit).
Evaluation and observability (Inspect AI, Braintrust, LangSmith, Langfuse, Phoenix, Logfire, OTel GenAI)
Borrow: Pydantic Evals for cases + datasets + LLMJudge + span-based evaluators (one dependency, matches your Pydantic stack); Logfire for zero-config Pydantic-native OTel tracing (two lines) or Phoenix for self-hosted (single Docker container); OTel GenAI experimental semantic conventions on all spans.
Reject (for now): Braintrust (SaaS, costs money, overkill for solo); LangSmith (LangChain-centric, per-seat pricing); Langfuse self-host (Postgres+ClickHouse+Redis+S3, disproportionate); Inspect AI (capability benchmarks, not production observability — revisit if you want to run HumanEval-style benchmarks).

Questions to answer before Milestone 1

Workflow version pinning on resume: hash match required, or warn-only, or ignored? Recommendation: hash required; --force-version-mismatch override.
AgentLoop subagent context isolation: fresh context per worker (Anthropic subagent model) or shared thread? Recommendation: fresh context is default; shared is opt-in with documented risk.

Budget cap threshold: $20/day? $10/run? Per-workflow override? Decide the policy now; implement in M1 or M2.
Fanout failure policy default: hard-stop, best-effort, or n-of-m? Recommendation: per-workflow, default hard-stop for modernization, best-effort for docs/review.
Ollama in M1 or M4? Recommendation: M4 or later. Simplifies M1-M3 significantly.
DAG in M1 or Pipeline-then-DAG? Recommendation: Pipeline in M1-M3, DAG in M4. Earns the complexity against a real second workflow.
Component contract for AgentLoop: is "weak determinism" defensible or should AgentLoop be marked as a distinct UnboundedComponent subtype? Recommendation: subtype it; name the looseness.

Does the two-machine setup actually save money at the user's volume? Quantify. If <$5/day, kill the complexity.

Closing note
The design is the work of someone who has thought carefully about what they're building. The component taxonomy is well-chosen, the layering is validated by AutoGen's independent arrival at the same split, and most of the conservative defaults (.. traversal blocks, allowed_executables, explicit context kwargs, no contextvars, hard retry floor) are correct for production reliability.
The risk is not that the architecture is wrong; it's that the architecture is premature. The fastest path to JVM modernization working is not a three-layer framework with a two-phase Planner and a DAG Orchestrator. It is a pydantic-ai Agent with three tools, a pydantic-graph for ordering, pydantic-evals for regression detection, and a YAML config loader — maybe 500 lines. The framework you're designing is the one that makes workflow #4 easy. Build it when workflow #3 has shipped and has told you what it actually needs. Until then, compose known-good libraries and keep the Primitives/Components split as a conceptual north star rather than a codebase requirement. That's the sequencing every framework in this comparison — LangGraph, LlamaIndex, Haystack, CrewAI — violated in its own way, and each is still paying the bill.
