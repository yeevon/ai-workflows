#### 1b. **Stdio-default invariant test** (M14-DA-05)

One assertion appended to `tests/mcp/test_http_transport.py` or added as a sibling file:

```python
def test_http_cli_default_transport_is_stdio() -> None:
    """Protect zero-flag MCP-host registrations (Claude Code / Cursor / Zed).

    A regression that flipped ``--transport`` default to ``http`` would
    silently break every existing stdio host registration. The stdio
    tests in ``tests/mcp/`` never shell through ``_cli``, so they don't
    catch this; the invariant has to pin the Typer option default
    explicitly.
    """
    transport_option = inspect.signature(_cli).parameters["transport"].default
    assert transport_option.default == "stdio", (
        "aiw-mcp --transport default must remain 'stdio' — "
        "M14 exit criterion 10 / backwards compatibility invariant"
    )
```

Pairs with the existing `test_http_default_bind_is_loopback` signature-level pin.

#### 1c. **HTTP/stdio schema-parity test for `run_workflow`** (M14-DA-SP)

New test in `tests/mcp/test_http_transport.py` (or new `tests/mcp/test_http_schema_parity.py` if the file grows):

- Invoke `run_workflow` over stdio (direct `tool.fn(payload)` — matches the existing `test_run_workflow.py` pattern).
- Invoke `run_workflow` over HTTP via `fastmcp.Client` on an ephemeral port (matches the existing T01 HTTP test pattern).
- Assert the two `RunWorkflowOutput` dicts are equal modulo fields that legitimately vary (`run_id` — autogenerated; possibly `started_at` if it's on the output surface; diff the full dict and enumerate the allowed-to-vary keys explicitly so a silent field addition fails).
- Use the same stubbed tier adapter for both invocations so the LLM output is deterministic and the comparison is meaningful.
- Pins KDR-008 (MCP schemas are public contract) at the transport layer — catches the regression class "FastMCP bumped a minor and now serialises pydantic models differently between stdio and HTTP."

#### 1d. **HTTP round-trip tests for `list_runs`, `cancel_run`, `resume_run`** (M14-DA-LR)

Three new tests (~30 LOC each) in `tests/mcp/test_http_transport.py` (or the new parity file):

- `test_http_list_runs_roundtrip` — seed two runs via `SQLiteStorage` (mirrors `tests/mcp/test_list_runs.py::_seed_run`), start the HTTP server, invoke `list_runs` via `fastmcp.Client`, assert the response shape contains both run_ids and `RunSummary`-compatible fields. No filter args needed — smoke coverage of the list-return serialisation path is the goal.
- `test_http_cancel_run_roundtrip` — seed a `pending` run, invoke `cancel_run` via `fastmcp.Client`, assert the response's `status == "cancelled"` and the Storage row flipped. Pins the envelope shape on the third tool.
- `test_http_resume_run_roundtrip_returns_envelope` — seed/run a workflow to gate pause via HTTP `run_workflow` (reuse the T01 pattern), then call `resume_run` over HTTP with an `approve` gate response, assert the returned envelope matches the stdio-side resume shape (status, plan, total_cost_usd, error). The deep resume-branch semantics (approve/reject/modify) are already pinned stdio-side; the HTTP version only needs to confirm reachability + envelope serialisation.

All five new tests reuse the existing `_reset_stub`, `_reensure_planner_registered`, `_redirect_default_paths`, `_reserve_ephemeral_port`, `_wait_for_http`, `_start_server` helpers — **no new fixtures required**.
